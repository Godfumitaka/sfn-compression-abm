## 1. 主張と非循環の規律（実装者が守る硬い規則）

### 1.1 一文（何の研究か）

> 圧縮という一つの機構の上で、類推の閾値・訂正フィードバック・誤差相関を動かすと、有用な抽象と共有誤信念が**同じ斜面に地続きで**出てくる。これを位相図（三象限が一枚の連続地図に乗り、条件を回すと一致から乖離へ連続変形する）という反証可能な形で示す。

殺す相手＝「正しい知識と集団誤信念は別の機構から来る」という素朴な二機構説。同一の圧縮機構が世界条件だけで両方を連続に生むなら、「ここまで科学・ここから神話」と引ける境界が地図のどこにもなくなる。**境界が引けない＝種類が違うの否定**。

### 1.2 実装者が壊してはいけない規律（最重要・違反すると全結果が無効）

以下は「あれば良い」ではなく「破れば実験が成立しない」硬い制約。新しいインスタンスは「不透明領域は別扱いが綺麗では」「ここで正解を補正すれば動く」と滑りやすい。滑ったら下の規律が止める。

1. **正解挙動を stipulate しない（§6 共通原理）**。各オペレータは「世界に何を置くか（`G_star`）」と「何が可観測か（`M`）」だけ決める。「この型ならこう振る舞え／この型は間違えろ」をコードに書いた瞬間、非循環性（関門B）が壊れる。誤るか正しく振る舞うかは閾値・コスト・可観測性の**掃引に内生**させる。
2. **leakage を構造で禁じる（§3）**。エージェント側コードは答え（`OracleView`）に**型レベルで到達できない**。自己申告 boolean では防げない。
3. **`if(target_type)` を書かない（§9）**。剪定・生成・採点・写像のどれも領域で分岐しない。ただし静的検査は**必要だが不十分**であり、本防御は §3 の構造分離である（§9.2）。
4. **二ゲインを合算しない（§7）**。`transfer_gain` と `coordination_gain` を足した量をどこにも作らない。
5. **送信は生インスタンスの無作為サンプリングのみ（§8）**。プロトタイプ経由送信は受け手の圧縮をバイパスし収束を出来レースにする。

## 3. 三層グラフと入力型の分離 ［Root Cause A・中心の修正］

### 3.1 三層グラフ

| 記号 | 定義 |
|---|---|
| `G_star` | 世界真理の完全関係グラフ（base 完全 ＋ target 完全） |
| `held_out_edge` | target 側で伏せる一辺（＝予測対象＝候補推論、§4）。`G_star` に含まれるが `AgentInput` からは除く |
| `M` | `M ⊆ G_star`。採点器がアクセスできる可観測部分集合 |

### 3.2 入力型の分離（旧版の全オペレータ `G = G_star` を廃止）

旧 `Instance` は `G_star`・`target_type`・`seed_id` を一つの構造に同居させ、leakage・ambient authority・`if(region)` 迂回を全部そこから派生させていた。これを型に物理分離する。**入力だけでなくエージェントの状態・設定・出力・フィードバックも型契約する**（v2の漏れ。状態が無契約だと oracle 由来値が state/closure/global/logger callback 経由で入りうる）。

```
# --- 公開入力（信頼済み redactor が G_star から生成する immutable DTO） ---
AgentInput  = {
    base_graph,              # source 役。完全に与える
    target_graph_partial,    # target 役。held_out_edge を除いたもの
    observable_mask,         # M を AgentInput に適用済み（§3.3）。公開辺の正の情報のみ
}
# --- エージェントの持ち物（すべて oracle-free） ---
AgentConfig = { threshold, correction_mode, lambda, ... }   # 掃引パラメタ。oracle-free
AgentState  = { prototype, public_history, rng_state }      # oracle-free
AgentOutput = { prediction, trace }                          # §6/§10
# --- オラクル側（検証専用） ---
OracleView  = { G_star, held_out_edge, target_type, seed_id }
ScoringKey  = { held_out_edge }                              # 評価器に渡す最小限
Feedback    = None | CorrectnessBit | RevealedEdge          # correction_mode ごとに明示
```

**実行は二フェーズ。フィードバックは予測確定後にしか渡さない。**

```
predict(AgentInput, FrozenAgentState, AgentConfig, RNG) -> (AgentOutput, PendingState)
update(PendingState, Feedback) -> AgentState
```

- **`AgentInput` は target_type・seed_id・G_star・held_out_edge を一切含まない**。`base_graph`/`G_star` との共有オブジェクト・back-reference・view・global registry を持たない immutable DTO とし、serialization round-trip を通す。`observable_mask` は完全グラフ上の bitmap や欠番ID列（伏せ辺の存在を逆算できる）ではなく、**公開済み辺についての正の情報だけ**を渡す。
- **エージェント側4モジュール（写像器・候補生成・MDL・通信器）は `AgentInput`・`AgentState`・`AgentConfig` のみを受ける**。`OracleView`・`Feedback`・`G_star` はこれらの**引数に存在しない**。
- `Feedback` に target_type・seed_id・完全な `G_star` を含めない。`correction_mode=transparent` でも渡すのは `CorrectnessBit` か（hold-out後の）`RevealedEdge` まで。
- redactor の位置（v2の C2 が逆だった点を修正）: **信頼済み redactor は `G_star` を読んで公開入力を作る**。`redact(G_star, held_out_edge, visibility_spec) -> AgentInput`。禁止すべきは redactor ではなく **redaction 後のエージェント側**。

### 3.2b 権限の分割（#2・採点の二義性を解消）

v2の「scorer が OracleView に触れてよい」は、予測前の内部仮説選択と予測後の正誤判定を混同していた。前者に oracle を許すと直接 leakage。権限を以下に割り、各々の入力で禁止対象を固定する。

| ロール | 入力 | oracle | 備考 |
|---|---|---|---|
| `ModelScorer`（内部MDL選択） | `AgentInput, AgentState` | **禁止** | 予測前。§5/§6 の `L(H)+L(D_obs\|H)` を計算 |
| `BaselinePredictor` | `AgentInput, frozen pre-oracle statistics` | **禁止** | 頻度統計は事前固定corpusか AgentInput のみ（§7.2） |
| `OracleEvaluator` | `AgentOutput, ScoringKey` | 許可 | 予測後。agent/state への参照禁止 |
| `QuadrantClassifier` | `MetricVector, preregistered_cutpoints` | **禁止** | target_type/seed_id を受けない（ラベル依存分類を断つ） |
| `Stratifier` | 分類後に target_type/seed_id を join | 許可 | 分析用。分類経路には入らない |
| `Logger` | evaluator 側の write-only sink | 許可 | §10 |

- `error_direction_prototype_consistent`（§6.4）等の oracle 診断は **post-hoc evaluator** が計算し、**エージェント状態へ戻さない**。
- 予測に影響する状態は prototype だけでない。§10 のスナップショットは **AgentState 全体**（prototype＋public_history＋rng_state）を対象にする。

- 保証は型レベル（混ざったら型エラー）。`committed_prototype_before_holdout` のような自己申告 boolean は証拠にしない（§10 で別途、不変スナップショットで担保）。

> これが停止級#1（hold-out辺が入力から除かれていない）・#6（ambient authority）・第二次#1（状態の無契約）・#2（採点権限の混同）への構造的回答。

### 3.3 可観測性の伝播 ［Root Cause B］

旧版は `M` を「採点器だけが見る集合」とし、エージェント入力は素通しだった。これだと「必要関係が見えないから誤る」がデータフローと一致しない（停止級#2）。修正:

- **`observable_mask` は `AgentInput` の側にも適用する**。役割反転の不可観測端（§6.3）で決定関係を消す場合、それを**採点器の `M` からだけでなく `AgentInput` からも除く**。
- ただし可観測性を入力まで伝播させると、決定関係を消した条件で**複数の束縛が観測同値（決定不能）**になりうる。この場合の誤りは「圧縮の失敗」ではなく「決定不能な穴を埋めた結果」である。層Aでの扱いは §7.4 と §6.3/§6.6 の注記に従う（神話象限の住所であり、共有プロトタイプで埋まるかが鍵。層Aでは収束を主張しないので、ここは層Bへの観測対象として記録のみ）。

## 4. 提示単位と hold-out ＝ 候補推論 ［#36・#4 の解決］

先行研究（Falkenhainer-Forbus-Gentner 1989）の正規手続きに合わせる。これにより transfer の意味が確定する。

- **提示単位は二グラフ**: source（base）と target を並べて与える。**base は完全に与え、target は一辺を伏せる**（原典で水流は「よく分かっているもの」、熱流が「問い」だったのと同じ配置）。
- **`held_out_edge` ＝ 候補推論（candidate inference）**: base 側にあり target 側に欠けている、共通構造につながった要素。transfer ＝ base→target の対応を通じてこの辺を当てられるか。
- **投影されるのは共通構造に接続した要素だけ**なので、高階の入れ子に伏せ辺が埋もれて漏れる問題（#4）は「消す」のではなく「target側で空ける」ことで回避される。`held_out_edge` は共通構造に接続した辺から選ぶ。
- **#36 の確定**: transfer ＝ **領域間の候補推論**。within-instance completion でも seed 分割でも entity 新規化でもない。
- 注: FFE は写像（mapping）のみを扱い、access と application は将来研究に残している。本ABMの鋳型適用ダイナミクス（§6.4 role_divergence）は**この上乗せ**であり、SMEから継承するものではない。

## 5. SMEエンジン境界 ［Root Cause C］

写像エンジン本体は再実装せず借用する。本書はその I/F を契約として境界に置く。ここが未契約だと共有述語名・oracle cleanup・手書き射影が「境界外」に逃げる（停止級#5/#8）。

### 5.1 I/F

```
map(base_graph, target_graph_partial) -> { alignment, total_score }
project(alignment, base_graph, target_graph_partial) -> Prediction
Prediction = EdgePrediction(edge_id) | Abstain(reason)   # 棄却を型で明示（§7.1）
```

- **写像器に渡すのは `AgentInput` の観測グラフのみ**。`target_type`・`seed_id`・`G_star`・`held_out_edge`・期待される対応表は渡さない。
- `alignment` ＝ 対応集合、`total_score` ＝ 構造評価スコア。
- `project` ＝ alignment から候補推論を投影する**固定・汎用規則**（共通構造に接続した未対応要素を投影。複数なら systematicity スコアで選ぶ）。**seed 別分岐を持たない**。構造一致が `threshold` に満たなければ `Abstain` を返す（§5.3・§7.1）。

### 5.3 圧縮目的関数と threshold の一般定義（#3・#4・全 operator 共通・一度だけ）

オペレータ節（§6.2/6.4）が「低 threshold→誤射影、高 threshold→棄却」と結果を書けてしまったのは、threshold が何に作用するかの一般式が無かったため。ここで一度だけ固定し、全 seed・全 operator で同一に使う。型別の判定規則を書くことを禁じる。

```
D_obs = AgentInput に実際に含まれる観測辺の全体（伏せ辺・非観測辺は単に D_obs に存在しない）
MDL(H) = L(H) + L(D_obs | H)             # 全 operator 共通。非観測辺を「採点不能」にしない
total_score(alignment) = 構造評価（systematicity 込み）
```

- **threshold が作用する単一の量**: `project` の候補受理は `total_score(best_alignment) >= threshold` のときのみ `EdgePrediction`、満たなければ `Abstain`。この一本だけ。`total_score`・MDL差・prototype activation を別々の場所で別単位で使わない。
- **codebook と符号長の単位**は operator 実装前に固定する（`L(H)`・`L(D_obs|H)` の述語あたり価格、変数・高階関係の符号化）。掃引で動かすのは threshold と lambda であって codebook ではない。
- **tie-break は全条件で一つだけ**。観測同値の候補が複数残るとき、prototype prior を使うなら **opaque や特定 seed だけでなく全条件の全観測同値候補に同じ規則で**適用する（種ごとに経路を変えない＝§6.0 遵守）。
- **オペレータは threshold・correction_mode・prototype 有無・lambda を受け取らない**。`PerturbationParams`（世界の作り方）と `AgentConfig`（掃引パラメタ）は別型（§3.2）。これにより threshold 腕ごとに刺激世界を変える実装が契約上排除される。同一 `instance_id` の `G_star` が全 threshold 腕で同一であることをテストする（§11）。

### 5.2 許される汎用バイアス / 禁じる課題固有知識

境界線＝**今回の seed・operator 名を知らなくても自然に書けるなら汎用機構。今回の課題配置を知っているから書けるなら答えの埋め込み**。

| 許す（汎用構造写像） | 禁じる（課題固有＝答えの埋め込み） |
|---|---|
| 1対1対応の優先 | `if target contains nucleus: map sun→nucleus` |
| 同 arity の述語を合わせる | `predicate_similarity("sun","nucleus")=high`（対象名の類似度） |
| 高階関係が保存される対応を優先 | `if predicates==[attracts,revolves]: solar_atom_template` |
| 整合的束縛・大きな共通構造を高評価 | `seed_id==0 ならこの mapping 候補を優先` |
| **tiered identicality**（同名述語は同一として一致） | role_divergence で「通常 mapping を作ってから反転候補を見る」 |
| **役割ベースのエンティティ対応**（対象名の似てなさに依らず役割で対応） | anti_analogy で「表層一致を一段落とす」 |

> **述語名の共有は漏洩ではない**。正規SMEは関係を同名どうしで合わせ（tiered identicality）、エンティティ（sun/nucleus）は**役割で**合わせる（名前の類似度は使わない）。よって「述語名を消す」は原典の機構を壊すので採らない。構造と表層を分ける本物の道具は systematicity（高階接続の選好）であり、その証拠は §7.2 の flat_matcher_baseline で取る。

## 6. 撹乱オペレータ（6種・改訂）

### 6.0 共通原理（再掲・最重要）

正解挙動を stipulate しない。各オペレータは `G_star`・`held_out_edge`・`M` の置き方だけ決め、`AgentInput`/`OracleView` を返す。全オペレータで `held_out_edge` は `AgentInput` から除かれ、`G_star`（その辺を含む）は `OracleView` に入る。**オペレータは `AgentConfig`（threshold 等）を受け取らない**（§5.3）。

> **§6.2/§6.4 の「低 threshold→…／高 threshold→…」は期待結果（反証対象）であって実装仕様ではない**。構造 pytest でこの誤答方向を assert しない（assert したらその run は非循環性の証拠にならない）。誤るか否かは §5.3 の単一 threshold 判定に内生する。

```
各オペレータ: (seed, PerturbationParams) -> (AgentInput, OracleView)
```

### 6.1 `isomorphic`（能力アンカー）
base 完全、target は一辺を伏せて与える、可観測。素の同型。測る量＝基準性能（ベースライン比の transfer_gain）。

### 6.2 `anti_analogy`（誤りサイト・表層誘惑＝ボトムアップ）
`G_star` に表層一致を足し（種2なら `diameter` を昇格）、高階前提を1本壊す。**壊れた関係は世界にあり可観測**。実視連星と同型。
*期待結果（反証対象・assert しない）*: 低 `threshold` で表層に釣られ誤射影（偽陽性）、高 `threshold` で破れを検出し棄却。

### 6.3 `role_reversal`（可観測性が操作軸）
非対称関係を反転（`asym(a,b)→asym(b,a)`、依存高階を再計算）。
- **可観測端**: 全構造可観測（能力アンカー）。
- **不可観測端**: 束縛決定関係を **`AgentInput` から除く**（採点器の `M` からだけでなく。§3.3）。
- **注（#2）**: 不可観測端は決定不能になりうる。エージェント個別のノイズで埋めれば各自バラバラ（ノイズ）、共有プロトタイプで埋めれば同方向（神話）。層Aでは収束を主張しないので、ここは「プロトタイプで埋まったか」を記録するのみ（§7.4）。

### 6.4 `role_divergence`（誤りサイト・鋳型同化＝トップダウン）＝**version (a)・全面改訂**

捕まえたい現象＝**見えている証拠を、身についた鋳型が上書きする。しかも鋳型の方向に間違える**（負の転移）。

```
require seed has >= 2 discriminating relations on one correspondence
target は事前固定の near-miss（エージェントの実行時プロトタイプを覗かない）
役割決定関係は target にあり可観測（隠さない＝隠したら opaque になる）
食い違わせる相手 = 原典の source が含意する正規の鋳型（設計時に既知）
held_out_edge = 役割決定関係に依存して向きが決まる辺（例: revolves_around）
```

- 旧版（実行時プロトタイプを覗いて衝突する `G_star` を後生成）は**廃止**。世界を外生に戻す（停止級#7解決）。
- **二腕で比較**: プロトタイプあり / プロトタイプなしのエージェント。差が出れば「原因は鋳型」と言える（停止級#24の時間交絡も同時に潰れる）。
- 誤りの向きが**鋳型と一致**したかを記録（§10 `error_direction_prototype_consistent`、§3.2b の post-hoc evaluator が計算）。「外した」では足りず「鋳型の向きに外した」が要る（停止級#21）。
- **層Aに残す**。
- *期待結果（反証対象・assert しない）*: 低 `threshold` で鋳型が局所証拠を上書き（あり腕が鋳型方向に誤射影）、高 `threshold` で局所証拠が勝つ。誤るか否かは §5.3 の単一判定に内生する。

### 6.5 `surplus`（制御・掃引外）
`G_star` に写像に噛まない不活性辺のみ足す。systematic な核だけ写像しノイズを無視できるかの一回確認。深さ選好の強度（競合囮）は試さない（FFE が証明済み、スコープ外）。

### 6.6 `opaque`（誤りサイト・被覆駆動）
役割決定関係を**採点器から隠す**（`M = G_star - {disc_pred}`）。これは「`disc_pred` を `D_obs` に含めない」だけであって、**残りの観測辺に対する `L(D_obs|H)` は通常どおり計算する**（§5.3）。一辺が尤度項を持たないことと全データ尤度をゼロにすることは別物——後者は書かない。hold-out 時に `G_star` を明かし、被覆駆動の選択と世界真理のズレ＝**誤信念量**を `transfer_gain` の低下として測る。
- role_divergence との区別: role_divergence ＝決定関係が**可観測**（鋳型が証拠を上書き）、opaque ＝決定関係が **`D_obs` に不在**（被覆駆動）。
- **tie-break は opaque 専用にしない**（§5.3）。観測同値が残ったときの規則は全条件共通の一つ。種ごとに経路を変えない。`recoverability`（残り構造から束縛が一意に復元できるか）は `OracleView` 側の **post-hoc 診断値**に限定し、tie-break 経路の分岐には使わない。
- 層Aの opaque は**単一エージェントの低転移誤信念**を測る。その社会的固定（神話＝低転移×高協調）は層B。
- *期待結果（反証対象・assert しない）*: `disc_pred` が `D_obs` に無いと被覆削減だけが効き、薄広ノードが選ばれて誤信念量が出る。これは結果であって stipulation ではない。

## 7. 二ゲインと指標 ［transfer＝accuracy・棄却契約・ベースライン lift・coordination 出力一致式］

### 7.1 `transfer_gain`（世界追従・非循環）＝**改訂：accuracy のみ。棄却の扱いを定義**

予測は型で `Prediction = EdgePrediction(edge_id) | Abstain(reason)`（§5.1）。層Aの exact-completion では:

```
hit      = 1 iff prediction == EdgePrediction(edge_id) かつ edge_id == held_out_edge,  else 0
coverage = 1 iff prediction is EdgePrediction,  else 0   # Abstain は coverage=0
transfer_accuracy = mean(agent_hit)                       # = transfer_gain（縦軸）
```

- **棄却（Abstain）は hit=0 として扱い、分母から除かない**（除外すると刺激が選別される）。棄却の挙動を別に見たいときは `coverage` と `selective_accuracy = mean(hit | coverage=1)` を**別指標として**記録し、transfer_accuracy に混ぜない。
- **DL を転移の軸に混ぜない**。混ぜると「世界に当たってないから低い」のか「表現が高くつくから罰されて低い」のかが混ざり、示したい機構（圧縮）を結果（誤信念）の測り方に埋め込む循環になる。
- **`accuracy` と `description_length` は別カラムで記録し、事前に合算しない**（§10）。
- **DL は捨てない**: 複雑さ軸（横軸）として残す。関門C/D のピーク一致/ずれ、関門E のλ位相不変性はこの横軸上で行う。λ重み付き効用 `accuracy - λ·DL` は**象限分類には使わず、関門Eの感度分析でのみ**算出する。
- **非循環性**: プロトタイプは `held_out_edge` を見る前にコミット（§10 不変スナップショット）。当否を `OracleEvaluator` が事後に追認するのみ。

### 7.2 二ベースライン（一本でなく二本・試行単位で保存）

```
frequency_baseline   = 出現頻度のみで予測。頻度の取得元は事前固定 training corpus か AgentInput のみ。
                       評価trialの OracleView を数えない（ベースライン自体の leak 防止）
flat_matcher_baseline = 同名述語の局所一致のみ（tiered identicality）で写像し投影する。
                        高階接続（systematicity）を使わない平らなマッチャー
```

- ベースラインは派生 gain だけでなく**試行単位で predicted_edge と hit を保存**する（§10）。後から別定義へ再集計できるように。
- 集約は**比でなく差（lift）**で単位を揃える（比は baseline accuracy=0 で破綻するため使わない）:

```
frequency_lift   = mean(agent_hit - frequency_baseline_hit)
flat_matcher_lift = mean(agent_hit - flat_matcher_baseline_hit)
```

- **flat_matcher_lift > 0 で初めて systematicity（構造で写像した）の証拠**になる（停止級#5の正しい対処＝述語除去ではなくこの対照）。頻度ベースラインだけでは「述語名で解いた」を統制できない。

### 7.3 `coordination_gain`（協調・世界を介さない）＝**改訂：出力カテゴリ一致に統一（#5）**

v2の式は左項（方向性のある予測精度）と右項（二者の一致確率）が別の量で、かつ正誤の二値に潰すと**どの誤答を共有したかが消え**、正答率を使う時点で OracleView が要り「世界を介さない」軸でなくなっていた。**正誤二値でなく、実際の出力カテゴリ（canonicalized された予測辺、または Abstain）の一致**に統一する。`predict_next_output_accuracy` 系の方向性予測は**併用しない**（混ぜると独立性が壊れる）。

```
y ∈ 出力カテゴリ = { canonicalized predicted_edge } ∪ { Abstain }   # 正誤ではない
observed_agreement = (1/N) Σ_i 1[ y_{A,i} == y_{B,i} ]
chance_agreement   = Σ_y p_A(y) · p_B(y)
coordination_gain  = observed_agreement - chance_agreement
```

- 周辺分布 `p_A, p_B` は**事前指定した同一 instance／condition strata 内**、または同じ刺激構成を保存する permutation null から推定する。**OracleView を使わない**（世界非依存を保つ＝transfer 軸との独立性）。
- これにより、二者が**同じ誤答辺を共有**したとき `coordination_gain > 0`（神話象限の縦軸として機能）。独立に多カテゴリへ散る誤答なら `≈ 0`。v2の二値式（0.68 を引く）は共有誤答を 0 に潰していた。

### 7.4 象限分類（和でなくベクトル）＋ error_correlation の身分

| transfer | coordination | 意味 |
|---|---|---|
| 高 | 高 | 有用な抽象 |
| 低 | 高 | 神話 |
| 高 | 低 | 孤立した正しい洞察 |
| 低 | 低 | ノイズ |

- `(transfer_gain, coordination_gain)` を常にベクトルでlog。一次元に畳まない。
- **`error_correlation` は層Aで数値実現しない**。それを実現する機構（系統異質性注入）は層B（付録Z）。層Aは `error_correlation` をラベルとしてlogするが、収束・神話固定を主張しない。**よって層Aの smoke-run は転移機構（§6 の各型を単一/同質エージェントで）を測り、社会的収束は測らない**（停止級#9・#26 へのスコープ宣言）。

## 8. 収束の操作化（層B・層Aでは主張しない）

収束は二本で測る。**ただし系統異質性注入（層B）が入って初めて意味を持つ**。層Aの smoke-run では走らせるが収束の主張はしない。両者とも **§7.3 の出力カテゴリ一致**の形式（正誤二値でなく実際の出力 `y`、chance は `Σ p_A(y)p_B(y)`、OracleView 不使用）に揃える。

```
error_agreement      = §7.3 の observed_agreement - chance_agreement を、誤答カテゴリ上で評価
pair_predictability   = §7.3 と同一の量（= coordination_gain）
```

伝達on/offは統制（収束が伝達由来か分布由来かの切り分け）。送信ポリシー(a)厳守（§9-C5）。

## 9. 禁止条項（機械的検査）＋ その限界

### 9.1 条項

- **C1** 離散バックエンド限定（連続埋め込み・勾配学習は層Aで使わない）。
- **C2** leakage防止（**向きを修正**）: 信頼済み `redact` は `G_star` を読んでよい。禁止対象は **redaction 後のエージェント側**——写像器・候補生成・MDL・通信器・`ModelScorer`・`BaselinePredictor` は `OracleView`・`G_star`・`Feedback`・`held_out_edge` を引数に取らない（型検査）。
- **C3** `if(region)`禁止: 剪定・生成・採点・MDL・**写像・前処理・alignment→prediction・通信buffer**のいずれも `target_type` で分岐しない。
- **C4** 二ゲイン合算禁止。
- **C5** 送信ポリシー(a)限定: 送信関数が prototype 内部表現（weights/ID）を参照しない（静的検査）。
- **C6** 非循環性: 共有述語名による正解の持ち込み・oracle cleanup・手書き近傍を禁止。プロトタイプは hold-out 前にコミット。`Feedback` は予測確定後にのみ渡す（§3.2）。
- **C7** 表層属性 allowlist: 表層一致に使える属性を明示列挙（種2では `diameter`）。メタ属性は使わない。
- **C8** near-miss判定は `total_score`。threshold 判定は §5.3 の単一量のみ。
- **C9** 二ベースライン同梱（§7.2）。未実装のまま `transfer_gain` を報告しない。頻度の取得元は事前固定corpusか AgentInput のみ（評価trialの OracleView 不使用）。
- **C10** ゲート付き smoke-run: 後続関門の指標は手前が緑になるまで読まない。
- **C11** 権限分割（§3.2b）: `QuadrantClassifier` は target_type/seed_id/OracleView を受けない。`OracleEvaluator` は agent/state を参照しない。oracle 診断値（`recoverability`・`error_direction_prototype_consistent`）はエージェント状態へ戻さない。
- **C12** オペレータ非汚染（§5.3）: オペレータは `AgentConfig`（threshold/correction_mode/lambda/prototype 有無）を受けない。同一 `instance_id` の `G_star` が全 threshold 腕で同一。

### 9.2 静的検査の限界（明示）

**C2/C3/C5/C6 の静的検査は必要だが不十分**。`target_type` を制御フローに出さなくても、辞書dispatch・クラス多相・関数ポインタ・配列index・前処理済みfeature（例: `prototype_required`、`len(G)-len(M)`、`seed_id`、特定述語の有無、提示位置）で地域を実質復元できる（停止級の迂回セクション）。

- **本防御は §3 の構造分離**（`AgentInput` が `OracleView` に型レベルで到達できない）。
- かつ **§7.4 のスコープ宣言**（層Aは機構・収束を主張しない）により、層Aで地域を復元しても得るものがない。
- 「6型を弁別可能に作る」（関門A）が潜在地域分類器を容易にする緊張は、設計思想の対立であって仕様の穴ではない。層を分離して押し返す（停止級#26）。

## 12. 関門Aの緑判定（改訂・#27）

**実装妥当性を合格条件にする。望ましい結果（ベースライン比で勝つ等）を合格条件にしない**（さもないと成功する刺激集合を選ぶまで設計を反復できる＝オーバーフィット）。

緑＝以下が**すべて**満たされる:
- 6つの `target_type` が、構造的に弁別可能な `(AgentInput, OracleView)` ペアを生成できる。
- **`AgentInput`/`AgentState`/`AgentConfig` が `OracleView`・`Feedback`・`G_star` に型レベルで到達できない**ことが検査で示される（§3.2・C2）。
- **権限分割（§3.2b）が型で成立**: `ModelScorer`/`BaselinePredictor` が oracle を受けず、`QuadrantClassifier` が target_type/seed_id を受けない（C11）。
- **threshold が §5.3 の単一量に集約**され、オペレータが `AgentConfig` を受けない（C12）。
- `frequency_baseline` と `flat_matcher_baseline` の両方が試行単位で実装されている（§7.2）。
- hold-out が実体のある候補推論である（§4）。`Prediction` 型で棄却を表現できる（§7.1）。

失敗条件（一つでも該当で未達）:
- 型を作り分けられない。
- エージェント側が `OracleView`/`Feedback`/`G_star` に到達できる。
- 採点・分類・ベースラインのいずれかが oracle やラベルを参照できる。
- threshold が複数の場所で別単位に分散している／オペレータが threshold を受ける。
- ベースラインが欠落、または棄却の扱いが未定義。

> 「ベースライン比で勝てるか」は**経験的結果（関門B以降）**であり関門Aの条件にしない。

緑を確認してから関門B（単一エージェントが非循環に有用な抽象を出す）へ進む。

## 付録Z: 層B据え置き項目（関門Cが緑になるまで実装しない）

## 付録M: 動機節への採用（実装に追加なし）

「真理化命題」を動機（ABMの外）として採用。実装には何も足さない。

> 抽象化は既存の共通性を抜き出す操作ではなく、同一でない領域を危険に同一視することから始まる。その同一視は出生時点で部分的に虚偽であり、後続の訂正・実践・合意によって写像の有効な部分集合が安定したとき有用な抽象となる。

これは `role_divergence`（正しい鋳型から識別を抜くと誤る）の時間反転として、低 `threshold` × 弱 `correction` の意義を言語化する。動機と証明対象の層は混ぜない。

## 付録E: v2→v3 改訂差分（第二次レッドチーム停止級6件への対処・レビュー用）

v2 への監査で返った停止級6件（leakage・非循環・指標定義）への対処。pro は「6件が反映され、新たな矛盾を生んでいないか」をここで照合する。性能の成否（割れるか・勝てるか・二腕差）は対象外。

1. **状態・設定・出力・フィードバックを型契約**（§3.2・第二次#1）。`AgentConfig`/`AgentState`/`AgentOutput`/`Feedback`/`ScoringKey` を新設。`predict→update` の二フェーズ化、フィードバックは予測確定後のみ。`redact` の位置を修正（C2 の禁止対象を redactor→redaction後のエージェント側へ）。スナップショットは prototype 単体でなく **AgentState 全体**。
2. **採点権限を6ロールに分割**（§3.2b・第二次#2）。`ModelScorer`（内部・oracle禁止）/`OracleEvaluator`（外部・状態禁止）/`QuadrantClassifier`（二ゲインのみ・ラベル禁止）/`BaselinePredictor`/`Stratifier`/`Logger`。C11 を追加。
3. **threshold の一般定義を §5.3 に一度だけ固定**（第二次#3）。作用する単一量＝`total_score >= threshold`。codebook 固定。オペレータは `AgentConfig` を受けない（`PerturbationParams` と別型・C12）。§6.2/6.4 の「低/高 threshold→…」を**期待結果（反証対象・assert禁止）**へ降格。
4. **opaque の stipulation 除去**（§6.6・第二次#4）。`L(D_obs|H)` を全 operator 共通にし、非観測辺は `D_obs` に不在なだけ（尤度ゼロ化しない）。tie-break は全条件共通の一つ。`recoverability` は post-hoc 診断に限定し経路分岐に使わない。
5. **`coordination_gain` を出力カテゴリ一致式に統一**（§7.3・第二次#5）。正誤二値（共有誤答を消す）と方向性予測（OracleView 依存）を廃し、`observed_agreement − Σ p_A(y)p_B(y)`（y＝予測辺/Abstain）へ。同じ誤答辺共有で `>0`。校正pytestに多カテゴリ・非対称 p を追加。
6. **transfer の棄却とベースラインを出力契約化**（§7.1/7.2・第二次#6）。`Prediction = EdgePrediction | Abstain`。`hit/coverage/selective_accuracy` 分離（棄却は hit=0、分母から除かない）。ベースラインは**試行単位で predicted_edge と hit を保存**、集約は**比でなく差（lift）**、頻度取得元は事前固定corpusか AgentInput のみ。

**pro が停止理由としなかった点（据え置き妥当）**: §4 候補推論＝hold-out、§6.4 外生 near-miss、§7.1 accuracy と DL 分離、二ゲイン非合算。構成比・勝敗・二腕差・象限分離は smoke-run 事項であり停止理由でない。
